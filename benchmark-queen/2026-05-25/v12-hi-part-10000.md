## Errors observed (queen log)

**Postgres deadlock during high-fanout push** — 0.12 hit a deadlock at `08:48:15.810Z`:

```
[error] [Worker 2] [libqueen] Query failed: ERROR:  deadlock detected
```

**49 push items affected**, all recovered via file-buffer failover:

| Event count | Severity | Message |
|---:|---|---|
| 1 | error | `Query failed: ERROR: deadlock detected` |
| 49 | warning | `PUSH: Database error detected: ERROR: deadlock detected` |
| 49 | info | `PUSH: Failed over 1 items to file buffer after DB error` |
| 1 | info | `Failover: Processing 49 events ... duplicate key errors are normal during recovery` |

**This is the file-buffer failover pattern working as designed** — the 49 affected pushes were persisted to disk during the deadlock and **fully recovered** when DB came back. The status endpoint reports `dbErrors: 1` for the visible event; **0 messages were lost**.

The same load on 0.14 produced **0 errors** — whatever was changed in the v3 push procedure (batch-INSERT statement order, advisory lock usage, or trigger refactoring) **eliminates this deadlock pattern** at 10 001 partitions.

## Headline takeaways

1. **Push throughput at 10 001 partitions: 17.3 k vs 21.0 k msg/s** — only ~18 % slower on 0.12, much smaller gap than the 2-partition test (where 0.12 was 54 % slower). Suggests **partition-scale routing was already good in 0.12**, but **partition-contention handling was bad**.
2. **Pop throughput collapses to 3.6 k msg/s** — same pattern as `v12-hi-part-1`. Whatever the pop-side improvement is between 0.12 → 0.14, it wins big in both extremes (high-contention 2-partition AND wide-fanout 10 001-partition).
3. **PG deadlock fired** during the test — and **the file-buffer failover saved every affected message**. This is the durability story working in practice: 1 deadlock event → 49 messages buffered to disk → 49 messages recovered. **0 data loss**.
4. **0.14 doesn't deadlock here** — the v3 push procedure rewrites must have improved lock ordering or trigger semantics. The `LIBQUEEN_TUNING.md` mention of "single INSERT so the trigger fires once with the full batch" is probably the relevant change.
5. **Pop batch efficiency is 2.79** (lowest seen anywhere) — the consumer essentially gets 3 messages per pop request. With 100 connections trying to drain 10 001 partitions on a struggling pop path, fan-out collisions dominate.
